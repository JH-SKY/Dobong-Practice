# 음악 스트리밍 서비스를 위한 데이터 모델링 (Music Data Modeling)

> **상태: 🚧 설계 진행 중 (Draft)**
> 본 프로젝트는 기초적인 1:N 관계를 넘어, 실제 서비스에서 발생할 수 있는 복잡한 데이터 관계를 정규화하고 구조화하는 과정을 담고 있습니다.

### 1. 기획 의도 (Why)
- 데이터 중복을 최소화하고 무결성을 유지하기 위한 **DB 정규화(Normalization)**의 원리를 체득함.
- 노래, 가수, 앨범, 소속사 등 음악 서비스의 핵심 엔티티 간의 유기적인 관계를 설계함.

### 2. 설계 구조 (ERD Architecture)
![Music_ERD_Draft](./Song_ERD_1차.png)
*(이미지 위치: /데이터베이스/Song_ERD_1차.png)*

- **주요 엔티티(Entity)**:
  - **Song (노래)**: 제목, 가사 정보를 포함하며 앨범 및 장르와 연결됨.
  - **Singer (가수)**: 가수명, 생년월일 정보를 포함하며 소속사와 1:N 관계를 가짐.
  - **Album (앨범)**: 발매일, 재킷 이미지 등을 포함하며 가수 및 소속사와 연결됨.
  - **Entertainment_Company (소속사)**: 여러 가수를 관리하는 최상위 조직.

### 3. 핵심 설계 포인트 (Current Progress)
- **다대다(N:M) 관계의 해결**: 
  - 한 곡에 여러 가수가 참여(피처링 등)하거나, 한 가수가 여러 곡을 부르는 관계를 해결하기 위해 **`Song_Singer`** 매핑 테이블을 도입함.
  - 앨범과 장르 사이의 관계 역시 **`Album_Genre`** 테이블을 통해 유연하게 설계함.
- **데이터 정규화**: 가수 정보와 소속사 정보를 분리하여, 소속사 정보 변경 시 가수 테이블을 일일이 수정하지 않아도 되는 구조를 구축함.

### 4. 트러블슈팅 및 고민 사항 (Ongoing)
- **문제**: 가수가 그룹인 경우와 솔로인 경우를 어떻게 하나의 테이블에서 효율적으로 관리할 것인가에 대한 구조적 고민 진행 중.
- **진행 상황**: 현재는 개별 Singer 위주로 설계되었으나, 향후 'Group' 엔티티를 추가하여 가수와의 포함 관계를 정의할 예정임.

### 5. 향후 계획 (Next Steps)
- 실제 SQL 스키마(`schema.sql`)로 변환하여 로컬 DB에 구축 시도.
- 연체료, 예약 등 도서관 시스템 설계에서 배운 로직을 확장하여 '음악 이용권 및 결제' 테이블 설계 추가.

### 6. 성장 포인트 (Retrospective)
- 단순히 표를 만드는 것을 넘어, 현실의 복잡한 비즈니스 로직을 데이터의 관점에서 쪼개고 연결하는 '아키텍처적 사고'를 연습함.
